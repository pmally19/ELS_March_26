DUNNING LEVEL CALCULATION ✅ CORRECTLY IMPLEMENTED
────────────────────────────────────────────────────────────────────────────

Algorithm (server/routes/finance/dunning-routes.ts:342-354):

  INPUT: days_overdue, procedure configuration
  OUTPUT: dunning_level (1-4)

  if days_overdue >= procedure.legal_action_days:
    level = 4 (Final/Legal Action)
  else if days_overdue >= procedure.final_notice_days:
    level = 3 (Final Notice)
  else if days_overdue >= procedure.level3_days:
    level = 3 (Third Notice)
  else if days_overdue >= procedure.level2_days:
    level = 2 (Second Notice)
  else if days_overdue >= procedure.level1_days:
    level = 1 (First Notice)
  else:
    level = 1 (default)

Example with default procedure:
  Days 1-6: Not eligible yet
  Days 7-13: Level 1
  Days 14-20: Level 2
  Days 21-29: Level 3
  Days 30-59: Level 3 (Final Notice)
  Days 60+: Level 4 (Legal Action)

AMOUNT CALCULATION ✅ CORRECTLY IMPLEMENTED
────────────────────────────────────────────────────────────────────────────

Formula (server/routes/finance/dunning-routes.ts:357-359):

  outstanding_amount = SUM(invoice.outstanding_amount)
  
  interest_amount = outstanding_amount 
                    × (interest_rate ÷ 100) 
                    × (days_overdue ÷ 365)
  
  dunning_amount = outstanding_amount 
                   + interest_amount 
                   + dunning_fee

Example:
  Outstanding: $10,000
  Days overdue: 30
  Interest rate: 12% annual
  Dunning fee: $25
  
  Interest = $10,000 × (12/100) × (30/365) = $98.63
  Dunning amount = $10,000 + $98.63 + $25 = $10,123.63

OVERDUE DETECTION LOGIC ✅ IMPLEMENTED
────────────────────────────────────────────────────────────────────────────

Source: billing_documents table
Query (server/routes/finance/dunning-routes.ts:312-316):

  SELECT customer_id, SUM(outstanding_amount), MAX(due_date)
  FROM billing_documents
  WHERE outstanding_amount > 0
    AND due_date < CURRENT_DATE
    AND posting_status IN ('POSTED', 'OPEN')
  GROUP BY customer_id
  HAVING SUM(outstanding_amount) >= minimum_amount

Logic:
  ✅ Considers only POSTED or OPEN invoices
  ✅ Ignores fully paid invoices (outstanding = 0)
  ✅ Respects minimum amount threshold
  ✅ Uses current date for overdue calculation
  ✅ Groups by customer (all overdue invoices together)

-- Query 4: Check overdue invoices
SELECT COUNT(*) as overdue_count,
       SUM(outstanding_amount) as total_overdue
FROM billing_documents
WHERE outstanding_amount > 0
  AND due_date < CURRENT_DATE
  AND posting_status IN ('POSTED', 'OPEN');

-- Query 5: Overdue breakdown by days
SELECT 
  CASE 
    WHEN CURRENT_DATE - due_date BETWEEN 1 AND 30 THEN '1-30 days'
    WHEN CURRENT_DATE - due_date BETWEEN 31 AND 60 THEN '31-60 days'
    WHEN CURRENT_DATE - due_date BETWEEN 61 AND 90 THEN '61-90 days'
    ELSE '90+ days'
  END as age_range,
  COUNT(*) as count,
  SUM(outstanding_amount) as amount
FROM billing_documents
WHERE outstanding_amount > 0
  AND due_date < CURRENT_DATE
  AND posting_status IN ('POSTED', 'OPEN')
GROUP BY age_range
ORDER BY age_range;

-- Query 6: Customers with dunning procedures assigned
SELECT COUNT(*) as with_procedure
FROM erp_customers
WHERE dunning_procedure IS NOT NULL
  AND dunning_procedure != '';
